# 第一部分：项目总览

## 1.1 项目名称

**秦岭火情监测系统** (Qinling Fire Monitoring System v1.0.0)

## 1.2 项目目标

构建一个面向秦岭山区的智能化森林火灾预警平台，集成 AI 图像识别、多源传感器数据融合、综合风险评估、自动告警、指挥大屏展示于一体，实现从"检测→评估→告警→处置→归档"的完整火灾预警闭环。

## 1.3 项目背景

秦岭是中国南北地理分界线，生态价值极高。近年来，森林火灾频发，传统人工巡检效率低、响应慢。本项目旨在利用深度学习和物联网技术，构建一套低成本、可部署、智能化的火灾预警系统，提升秦岭山区的防灾减灾能力。

## 1.4 项目解决的核心问题

| 问题 | 解决方案 |
|------|---------|
| 人工巡检效率低 | CNN模型自动识别火灾/烟雾，秒级响应 |
| 传统报警延迟大 | 风险评分自动触发告警，无需人工干预 |
| 决策缺乏数据支撑 | 传感器+AI融合评分，提供量化决策依据 |
| 信息展示不直观 | 指挥大屏 + Leaflet地图 + 实时数据流 |
| AI不透明不可信 | Grad-CAM热力图可视化 + 模型评估页面 |
| 告警管理混乱 | 完整告警生命周期：pending→acknowledged→resolved |
| 部署维护复杂 | Docker容器化 + 一键启动脚本 |

## 1.5 目标应用场景

- **林业防火部门**：日常森林火情监控
- **应急指挥中心**：灾害发生时的指挥大屏
- **科研教学**：AI+IoT融合的示范系统
- **比赛展示**：综合性强、可视化突出的完整系统

## 1.6 比赛定位

本项目定位为 **"AI+物联网+可视化"三位一体的综合应用系统**，不是单纯的算法项目，而是一个**工程化的平台级产品**。

## 1.7 系统核心价值

1. **AI可信性**：Grad-CAM热力图让AI决策"看得见"
2. **多源融合**：AI视觉 + 传感器数据的综合风险评估
3. **完整闭环**：检测→评估→告警→确认→解决的完整生命周期
4. **工程化交付**：Docker、一键启动、健康检查、日志系统
5. **视觉冲击力**：指挥大屏的科技风设计

## 2.3 数据流向

```
图片上传 → 前端app.js → POST /detect → api_server.py
    → fire_detector.detect_single()  [AI推理]
        → 模型预测: class, confidence, probabilities
        → Grad-CAM: heatmap (base64)
        → 内置风险评估: score, level
    → risk_assessor.assess_risk()  [综合评估]
        → AI风险(60%) + 传感器风险(40%)
    → database.save_detection()  [持久化]
    → 返回JSON给前端

前端接收结果 → 更新UI:
    → 显示分类结果 + 置信度
    → 叠加Grad-CAM热力图
    → 更新风险等级标签
    → 更新概率环形图
    → 更新风险趋势折线图
    → 检查是否触发告警
        → 高风险自动调用 POST /send-alert
        → 后端保存告警记录
        → 前端显示告警横幅

传感器数据流:
    后端虚拟传感器 → GET /sensors → 前端每10秒轮询
    → 显示温度/湿度/风速/空气质量
    → 参与综合风险评分
```

## 2.4 模块依赖关系

```
config.py ← fire_detector.py ← api_server.py ← 前端app.js
                                    ↑
                              database.py (读写)
                              risk_assessor.py (评估)
                              sensor_integration.py (数据)
```

- **config.py** 是所有模块的基础配置，提供路径、参数、阈值
- **fire_detector.py** 依赖config，负责AI推理和Grad-CAM
- **api_server.py** 是中枢，调用detector、assessor、sensor、database
- **database.py** 独立模块，被api_server调用
- **前端** 只与api_server通过HTTP交互，不直接访问后端模块

## 2.5 核心运行流程

1. 用户启动：`start.bat` → `python run.py` → 启动Flask服务(8001) + Nginx(80)
2. 浏览器访问：`http://localhost` → Nginx代理到前端静态文件
3. 前端初始化：检查后端连接 → 加载检测历史 → 启动传感器轮询
4. 用户上传图片 → 前端发送到后端 → AI推理 + 风险评估 → 返回结果
5. 高风险自动触发告警 → 告警生命周期管理
6. 指挥大屏可独立展示：地图 + 数据流 + 趋势图

## 2.6 页面之间关系

| 页面 | 功能定位 | 数据来源 | 独立性 |
|------|---------|---------|--------|
| dashboard.html | 日常监控操作 | 后端API | 依赖后端 |
| dashboard_large.html | 指挥展示大屏 | 后端API + 本地模拟 | 可独立运行 |
| evaluation.html | 模型评估展示 | 内置静态数据 | 完全独立 |

三个页面互不干扰，可同时打开。评估页面完全独立，不依赖后端。

## 2.7 API调用关系

| API接口 | 方法 | 调用方 | 核心功能 |
|---------|------|--------|---------|

| `/alert-history` | GET | 前端查询 | 告警列表 |

## 2.8 数据库与业务关系

**detections表** ← 每次检测自动写入，支持历史查询和趋势分析
**alerts表** ← 高风险检测自动/手动创建，支持生命周期管理
**sensors表** ← 传感器数据持久化，支持历史回放

三表通过时间戳关联，detections和alerts通过detection_id关联。

```

    → 后端: AI推理 + 风险评估 + 数据库保存
    → 返回: class, confidence, probabilities, heatmap, riskAssessment

### 3.1.4 告警流程

```
检测完成 → checkAndTriggerAlert(result)

        → 后端保存到alerts表 → 返回alert_id
        → 前端显示告警横幅 + showAlert()
    → NO: 隐藏手动告警按钮

# 第四部分：后端系统分析

## 4.1 api_server.py 详细分析

### 整体架构

Flask单文件应用，约800行代码，采用装饰器路由，集成所有业务模块。

### API接口逐个分析

| 接口 | 方法 | 输入 | 输出 | 功能 | 调用方 | 系统作用 |
|------|------|------|------|------|--------|---------|
| `/health` | GET | 无 | `{status, timestamp}` | 健康检查 | 前端初始化 | 服务可用性探测 |
| `/detect` | POST | `file`(图片) | `{class, confidence, heatmap, riskAssessment, probabilities}` | AI检测+风险评估 | 前端检测按钮 | **核心接口** |
| `/sensors` | GET | 无 | `{temperature, humidity, wind_speed, air_quality}` | 传感器数据 | 前端10s轮询 | 环境数据采集 |
| `/assess-risk` | POST | `{detection_result, sensor_data}` | `{score, level, description}` | 综合风险评估 | 前端(备用) | 独立评估入口 |
| `/send-alert` | POST | `{risk_level, risk_score, detection_type, location}` | `{alert_id, status}` | 创建告警 | 前端自动/手动 | 告警生成 |
| `/alert-history` | GET | 无 | `[{id, alert_id, status, ...}]` | 告警列表 | 前端查询 | 告警管理 |
| `/alert/acknowledge` | POST | `{alert_id, acknowledged_by}` | `{success}` | 确认告警 | 前端操作 | 告警生命周期 |
| `/alert/resolve` | POST | `{alert_id}` | `{success}` | 解决告警 | 前端操作 | 告警生命周期 |
| `/history` | GET | `?limit=50` | `[{id, prediction, confidence, ...}]` | 检测历史 | 前端初始化 | 历史回溯 |
| `/system-status` | GET | 无 | `{model_loaded, database_connected, today_detections, ...}` | 系统状态 | 前端30s轮询 | 运维监控 |
| `/risk-trend` | GET | `?hours=24` | `[{hour_bucket, avg_score, count}]` | 风险趋势 | 大屏页面 | 数据可视化 |
| `/stats` | GET | 无 | 统计数据 | 综合统计 | 大屏页面 | 数据可视化 |
| `/evaluation-data` | GET | 无 | 模型评估数据 | 评估数据 | 评估页面 | AI可信性 |

# 第五部分：AI能力分析

## 5.1 当前AI模型作用

- **模型类型**：轻量级CNN（3层卷积 + 2层全连接）
- **输入**：64×64 RGB图像
- **输出**：4分类概率（fire/smoke/fire_smoke/normal）
- **模型大小**：约几百KB~几MB
- **推理速度**：CPU下约100-500ms

## 5.2 风险评分逻辑

系统存在**两套风险评估**：

1. **fire_detector内置评估**：基于类别+置信度的简单公式
2. **risk_assessor综合评估**：AI(60%) + 传感器(40%) 的加权融合

最终使用综合评估结果，更具说服力。

## 5.3 Grad-CAM可信度设计

- 找到最后一个卷积层
- 计算目标类别相对于特征图的梯度
- 全局平均池化得到通道权重
- 加权组合生成热力图
- JET颜色映射 + 与原图叠加
- 失败时回退到简单高斯热力图

**这是比赛的核心亮点之一**：让评委看到"AI在看哪里"。

## 5.4 模型评估页面意义

- 混淆矩阵：证明分类准确性
- 分类报告：Precision/Recall/F1三个维度
- 训练曲线：Loss收敛+Accuracy提升
- **目的**：建立AI可信性，回答"凭什么相信AI"的问题

## 5.5 当前AI部分的真实强度

**优势**：
- 完整的训练→推理→可视化流程
- Grad-CAM提供可解释性
- 4分类覆盖主要火灾场景
- 集成风险评估，不是孤立的分类器

**不足**：
- 64×64分辨率偏低，真实场景可能不够
- 数据集规模有限（增强后约5000张）
- 模型轻量但准确率上限受限
- 未使用预训练模型（如ResNet、EfficientNet）
- Grad-CAM在64×64下的解释力有限

## 8.1 技术短板

| 短板 | 严重程度 | 说明 |
|------|---------|------|

| 前端无框架 | 低 | 原生JS，维护成本高但性能好 |

## 8.2 仅限演示级的功能

1. **虚拟传感器**：数据基于时间戳确定性生成，非真实采集
2. **告警发送**：`/send-alert`只保存数据库，没有真正通知消防部门
3. **地图标记**：大屏地图上的标记是预设的，非GPS实时定位
4. **用户系统**：无注册/登录，确认告警人固定为"admin"

## 8.3 模拟数据清单

- 传感器温度/湿度/风速/空气质量 → 虚拟生成
- 告警"已通知消防" → 仅模拟文字
- 大屏地图坐标 → 预设坐标
- 系统运行时间 → 简单计算

## 8.4 不适合生产环境的点

1. SQLite不支持高并发写入
2. 无认证鉴权，接口裸露
3. 无HTTPS，数据明文传输
4. 无负载均衡，单进程Flask
5. 无监控告警（对系统本身的监控）
6. 无数据备份策略
7. temp文件未自动清理（后端存在temp_*.jpg）

## 8.5 未来必须重构的模块

1. **api_server.py**：拆分为多个Blueprint/模块
2. **传感器系统**：抽象接口，支持真实硬件接入
3. **前端**：考虑引入Vue/React提升可维护性（如果项目规模扩大）
4. **数据库**：如果并发增大，迁移到PostgreSQL

## 8.6 性能瓶颈

- Flask单进程，无法利用多核
- 每次检测加载图片到内存，高并发时内存压力大
- SQLite写锁，大量并发写入时性能下降
- 前端轮询间隔固定，无法根据负载动态调整

## 8.7 安全边界

- 文件上传仅检查扩展名，未检查文件内容（magic bytes）
- 无请求频率限制
- 无输入验证（检测接口）
- 无API密钥认证

## 8.8 架构上限

当前架构适合：单机部署、小规模使用（<50并发）、比赛演示

不适合：大规模生产、多节点部署、高并发场景

| "这个系统有人在用吗？" | ⭐⭐⭐ | 展示系统能力和部署方案，而非声称已在使用 |

#### 3. 最需要补强什么

1. **模型评估数据**（最重要）：没有precision/recall/F1，AI部分就是"黑箱"
2. **真实数据集**：哪怕是小规模的真实森林火灾图片也好
3. **部署演示能力**：Docker一键启动，现场演示
4. **安全基本功**：至少有输入校验和文件限制

#### 4. 最适合展示的部分

#### 5. 应该弱化的部分

- 传感器系统（容易被质疑"假数据"）
- 模型架构细节（64×64分辨率会暴露问题）

#### 6. 项目"故事线"是否成立

**成立，但需要调整叙事角度**。

❌ 不要说："我们做了一个火灾检测AI模型"（模型太简单，撑不住）
✅ 要说："我们构建了一个**AI驱动的森林火情智能监测与响应系统**，实现了从图像感知→风险量化→自动告警的完整事件响应链路"

**核心叙事**：重点不是AI模型有多强，而是**AI如何被工程化地整合到业务流程中**。展示的是"系统思维"而非"模型思维"。

### 虚拟消防告警模块对系统架构的影响

#### 架构演化分析

**告警模块的加入标志着项目从"AI识别Demo"向"事件响应系统"的关键演化**。

**演化前的架构**：
```
输入 → AI处理 → 输出展示（单向数据流）
```

**演化后的架构**：
```
输入 → AI处理 → 风险评估 → 阈值判断 → 事件响应 → 历史记录
                    ↑                              │
                    └──────── 反馈循环 ─────────────┘
```

这是一个**质的变化**：
- **从"工具"到"系统"**：不再只是"识别一下告诉你结果"，而是"识别→判断→行动→记录"
- **从"被动"到"主动"**：不再等待用户查看结果，而是主动触发告警
- **从"无状态"到"有状态"**：有了告警历史、去重机制、事件ID

#### 对后续架构演化的奠基作用

告警模块建立了以下**架构基础**，后续可以在此基础上扩展：

1. **事件触发机制**：阈值判断逻辑可以扩展为更复杂的规则引擎
2. **去重机制**：基于image_id的去重，可以扩展为基于时间窗口、基于地点的去重
3. **告警历史**：内存存储可以无缝替换为数据库存储
4. **告警通道抽象**：当前的模拟发送，可以替换为真实的短信/邮件/Webhook
5. **前端告警UI**：横幅+历史面板的UI模式，可以扩展为完整的告警管理中心

#### 总体判断

**虚拟消防告警模块虽然功能上是"模拟"的，但架构上是"真实"的**。它为系统建立了事件驱动的骨架，后续只需往骨架上填充真实能力（持久化、真实通知、确认机制），而不需要重新设计架构。这是一个**正确的、低成本的架构投资**。
